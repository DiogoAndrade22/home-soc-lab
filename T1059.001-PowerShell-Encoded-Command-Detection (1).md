## 1. Executive Summary

A simulated attack technique was executed against a monitored Windows 11 endpoint in an isolated home SOC lab, and was successfully detected by the SIEM. The technique — abuse of PowerShell to run a **base64-encoded command** — is a common method real attackers use to obscure malicious activity from defenders inspecting command lines.

The activity was generated using Red Canary's **Atomic Red Team** framework (test `T1059.001-14`), captured by **Sysmon**, forwarded by the **Wazuh agent**, and raised as a **high-severity (level 12) alert** by the Wazuh manager, correctly mapped to **MITRE ATT&CK technique T1059.001**.

This report documents the emulation, the resulting detection, the supporting evidence, and the analytical separation of the true signal from coincidental lab noise.

## 5. MITRE ATT&CK Mapping

| Tactic | Technique | ID |
|---|---|---|
| Execution | Command and Scripting Interpreter: PowerShell | T1059.001 |

**Why this matters:** Encoded PowerShell (`-EncodedCommand` / `-EA`) is a hallmark of "living off the land" attacks — adversaries use trusted, built-in Windows binaries to evade detection. Base64 encoding obscures the payload from defenders who scan command lines for known-bad strings. The combination flagged here (a WMI-spawned PowerShell process running encoded content) is a recognised attacker pattern, which is why it warrants a high severity score.

## 7. Findings & Analyst Notes

**Primary finding (true positive — by design):** The single **rule 92071** alert is the detection that corresponds to the emulated technique. It captured the process, the encoded command line, and the parent-process lineage, and mapped the activity to the correct ATT&CK technique. The parent process was confirmed as `WmiPrvSE.exe`, corroborating the rule's WMI-spawned classification.

**Signal vs. noise:** The dashboard also showed a large volume of **rule 92213** (level 15, "Executable file dropped in folder commonly used by malware") alerts around the same window. These are assessed as not directly part of the encoded-command test — they most likely originated from the AtomicTestHarnesses module installation and other Atomic Red Team file artifacts written to disk during setup.

## 8. Detection & Hardening Notes (Defender Perspective)

If this were a real environment, recommended follow-ups would include:

- **Alert on encoded PowerShell** as a high-priority rule (already covered by Wazuh 92071) and surface the decoded content where possible.
- **Monitor WMI-spawned processes** (`WmiPrvSE.exe` as a parent) — a common execution and persistence vector.
- **Enable PowerShell Script Block Logging** (Event ID 4104) for deeper visibility into what encoded commands actually do at runtime, beyond the command line alone.
- **Constrained Language Mode / execution policy** controls to limit PowerShell abuse on endpoints that don't require full scripting.
